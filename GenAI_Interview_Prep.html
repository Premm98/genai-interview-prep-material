RAG — End to End Pipeline
RAG pipeline stages +

Full flow: Ingestion → Chunking → Embedding → Indexing → Query → Retrieval → Reranking → Augmentation → Generation → Post-processing

  • Ingestion: Load raw docs (PDF, HTML, DB, API). Use LangChain DocumentLoaders or LlamaIndex Readers.
  • Chunking: Split into meaningful units. Strategy determines everything downstream.
  • Embedding: Convert chunks to dense vectors using embedding model.
  • Indexing: Store in vector DB (Pinecone, Weaviate, ChromaDB, FAISS, pgvector) with metadata.
  • Query processing: Embed query, optionally rewrite or expand it first.
  • Retrieval: Top-k similarity search. Can combine dense + sparse (hybrid search).
  • Reranking: Cross-encoder re-ranks top-k for precision before sending to LLM.
  • Augmentation: Inject retrieved context into prompt template.
  • Generation: LLM generates final answer grounded in context.
  • Post-processing: Citation extraction, hallucination check, output formatting.
Chunking strategies — when to use what +
  • Fixed-size Simple, fast. Use for homogeneous docs. Risk: breaks semantic boundaries. Typical: 256–512 tokens.
  • Recursive character Splits paragraph → sentence → word. Best general-purpose default in LangChain. Use first.
  • Semantic chunking Embeds sentences, splits where cosine similarity drops. Best quality, slower. Use for mixed-format docs.
  • Sliding window Overlapping chunks (512 tokens, 50 overlap). Preserves context at boundaries. Higher storage cost.
  • Parent-child (hierarchical) Small chunks for precise retrieval, parent chunk returned to LLM for context. Best of both. LlamaIndex ParentChildRetriever.
  • Document-aware Markdown headers, HTML tags, PDF sections as natural split points. Best for structured content.

Rule of thumb: chunk size should match your query scope. Narrow Q&A → small chunks (256). Summarization → large chunks (1024).

Embedding models — how to choose +
  • text-embedding-3-large (OpenAI) Best general-purpose. 3072-dim. Use when quality > cost.
  • BGE-M3 (BAAI) Open source, multilingual, supports dense + sparse + colbert. Best open-source choice.
  • all-MiniLM-L6-v2 Lightweight, fast. Good for prototyping or latency-critical apps.
  • Cohere embed-v3 Strong for RAG, supports int8 compression, good multilingual support.
  • Domain fine-tuned Always best for specialized domains. Fine-tune using sentence-transformers on triplets (query, positive, negative).

Always evaluate on your actual retrieval task before committing to a model. Use MTEB benchmark or custom eval set.

Vector databases — comparison +
  • Pinecone Managed, scales well, fast. Best for production with no infra team. Expensive at scale.
  • Weaviate Open source + managed. Hybrid search built-in, GraphQL API, rich metadata filtering.
  • ChromaDB Dev-friendly, lightweight. Great for prototyping. Limited production features.
  • pgvector Postgres extension. Best if you already have Postgres. No extra infra. Limited ANN performance at scale.
  • FAISS Facebook library, in-memory or disk. Fastest for batch retrieval. No built-in metadata filtering.
  • Qdrant Rust-based, fast, good filtering, open source. Growing favorite for production self-hosted setups.
RAG — Reducing Latency
Retrieval-side latency fixes +
  • ANN index (HNSW) Use approximate nearest neighbor instead of exact search. 10–100x faster with minimal accuracy loss. Tune: ef_search, M, ef_construction.
  • Reduce embedding dimensions Use Matryoshka embeddings (OpenAI text-3) — truncate to 256 or 512 dims. 4x faster with ~2% quality drop.
  • Scalar quantization (int8) Compress float32 → int8. 4x memory reduction, 2–4x faster retrieval. Cohere embed-v3 natively supports this.
  • Binary quantization Extreme compression — 32x smaller. Use with oversampling + rerank to recover quality. Supported in Pinecone, Weaviate.
  • Metadata pre-filtering Filter by date, category, user_id before vector search. Dramatically reduces search space. High-cardinality filters in metadata, not vectors.
  • Semantic query cache Cache query embeddings + results for repeated/similar queries. Redis + GPTCache. Best for FAQ-style workloads.
  • Async parallel retrieval Multiple sources → run concurrently with asyncio.gather(). Never await sequentially if retrievals are independent.
Generation-side latency fixes +
  • Streaming responses stream=True — show tokens as they generate. UX feels instant even if TTFT is same. Critical for user-facing apps.
  • Smaller context window Reduce retrieved chunks. Each extra token costs time. Top-3 beats top-10 if reranking quality is high.
  • Smaller/faster model GPT-4o-mini or Claude Haiku for simple RAG QA. Reserve large models for complex synthesis. Route by query complexity.
  • Prompt compression Use LLMLingua to compress retrieved context 3–5x before LLM. Reduces tokens without losing key info.
  • KV / prompt caching Cache static system prompt + context prefix. Anthropic prompt caching saves 85%+ latency + cost on repeated prefixes.
  • Speculative decoding Small draft model generates tokens, large model verifies. 2–3x speedup. Groq hardware implements natively.
Architecture-level latency fixes +
  • Pre-compute embeddings Embed all docs at ingestion time. Only query embedding happens at runtime. Never embed on the fly.
  • Connection pooling Reuse vector DB connections. Cold connections add 50–200ms. Persistent clients, not per-request instantiation.
  • Colocate services Vector DB + LLM API in same cloud region. Cross-region adds 100–300ms easily.
  • Async end-to-end FastAPI async routes + async LLM calls + async DB queries. Never block the event loop.
  • Two-stage retrieval Fast coarse retrieval (BM25 or small model) → rerank with expensive model only top 20. Two-stage beats one expensive stage.
RAG — Improving Result Quality
Query-side improvements +
  • HyDE (Hypothetical Document Embeddings) LLM generates a hypothetical answer first, embed that instead of raw query. Dramatically improves retrieval for short/vague queries.
  • Query rewriting LLM rewrites user query to be retrieval-friendly. "What's the cap?" → "What is the maximum limit for the X feature in product Y?"
  • Multi-query retrieval Generate 3–5 query variants, retrieve for each, deduplicate. Broader coverage, catches synonyms and phrasings.
  • Step-back prompting Generalize the specific query to a broader question, retrieve on both. Helps when answer requires background context.
  • Query decomposition Break complex multi-part query into sub-questions. Answer each independently, combine. Good for analytical queries.
  • Query routing Classify query first (FAQ, analytical, procedural), route to different retrievers or prompts. Improves precision per query type.
Retrieval improvements +
  • Hybrid search Combine dense (semantic) + sparse (BM25/keyword). Use Reciprocal Rank Fusion (RRF) to merge. Handles both semantic and keyword matches.
  • Reranking Cross-encoders (Cohere Rerank, BGE Reranker, ColBERT) re-score top-k. 20–40% precision improvement. Always add this layer.
  • Contextual compression LLM extracts only relevant sentences from each retrieved chunk before generation. Removes noise.
  • Parent-child retrieval Retrieve small chunks for precision, pass parent chunk to LLM for full context. Best for long documents.
  • Metadata filtering Structured filters (date range, doc type, department) alongside vector search. Don't let vector do all the work.
  • Knowledge graph hybrid Combine vector retrieval with graph traversal. Captures entity relationships that dense vectors miss. Complex interconnected domains.
Generation improvements +
  • Citation grounding Instruct LLM to cite [chunk_id] for each claim. Enables verification, reduces hallucination.
  • Strong system prompt "Only answer using the provided context. If the answer is not in the context, say you don't know." Explicit instruction dramatically reduces hallucination.
  • Chain-of-thought Ask LLM to reason step-by-step before answering. Reduces errors on complex retrievals.
  • Iterative RAG / FLARE LLM generates answer, pauses when uncertain, triggers another retrieval, continues. Best for complex multi-hop questions.
  • Corrective RAG (CRAG) Evaluate retrieved docs — if low relevance, fall back to web search or broader corpus. Self-correcting pipeline.
RAG — Evaluation Metrics
Retrieval metrics
Precision@k
Of top-k retrieved chunks, what fraction are relevant? Measures retrieval precision.
Recall@k
Of all relevant chunks, what fraction were retrieved? Measures coverage.
MRR
Mean Reciprocal Rank. How high does the first relevant result appear? Best for single-answer scenarios.
NDCG@k
Normalized Discounted Cumulative Gain. Quality weighted by position. Best for ranked retrieval.
Hit Rate
Did at least one relevant chunk appear in top-k? Simpler than recall but still useful.
Generation metrics
Faithfulness
Is the answer supported by retrieved context? The key anti-hallucination metric. RAGAS measures this.
Answer Relevance
Is the answer relevant to the question? Catches responses that are faithful but off-topic.
Context Precision
Are the retrieved chunks actually relevant to answer the question? Diagnoses retrieval noise.
Context Recall
Does retrieved context contain all info needed? Diagnoses retrieval gaps.
ROUGE / BLEU
N-gram overlap with reference answer. Useful but limited — doesn't capture semantic equivalence.
BERTScore
Semantic similarity between generated and reference using BERT. Better than ROUGE for meaning.
Eval frameworks
  • RAGAS End-to-end RAG evaluation. Faithfulness, answer relevance, context precision/recall. LLM-based — no human labels needed. Most asked about in interviews.
  • TruLens RAG Triad: answer relevance + context relevance + groundedness. Good observability dashboard.
  • DeepEval Unit-test style for LLM outputs. Define metrics as assertions. Good for CI/CD integration.
  • LangSmith Tracing + evaluation for LangChain apps. Debug retrieval step-by-step in production.
RAG — Real Problems & Solutions
Problem 1
Retrieval returns wrong or irrelevant chunks
Query and document are in different semantic spaces. Embedding model not suited for domain. Short queries have weak embeddings.
Solutions
1. Add HyDE — embed hypothetical answer instead of raw query. 2. Fine-tune embedding model on domain-specific pairs. 3. Switch to hybrid search (BM25 + dense). 4. Add reranking to fix rank order. 5. Evaluate embedding model on your custom eval set.
Problem 2
LLM hallucinates despite correct context being retrieved
LLM ignores context and generates from parametric memory. Too much context confuses the model. Context buried in middle of long prompt (lost-in-the-middle problem).
Solutions
1. Stronger system prompt: "Only use the provided context." 2. Reduce context size — rerank, pass only top 3. 3. Put most relevant chunk first or last (not middle). 4. Contextual compression to remove noise. 5. Switch to model with better instruction following (Claude is better than GPT-4 for faithfulness).
Problem 3
Answer requires info from multiple chunks (multi-hop)
Single retrieval step doesn't capture cross-document reasoning. "What is A's relationship with B?" may need 3 separate retrieved facts combined.
Solutions
1. Iterative RAG / FLARE — retrieve, generate partial answer, retrieve again. 2. Query decomposition — break into sub-questions. 3. Knowledge graph — traverse entity relationships. 4. Agentic RAG — agent decides when to retrieve again.
Problem 4
Chunking breaks semantic units
Fixed-size chunking splits tables, code blocks, structured data mid-way. Retrieved chunk is meaningless without surrounding context.
Solutions
1. Document-aware splitting (markdown headers, HTML tags). 2. Parent-child retrieval — retrieve small chunk, return full parent. 3. Sliding window with overlap. 4. Semantic chunking to detect natural boundaries. 5. Special handlers for tables — extract to structured format before chunking.
Problem 5
Retrieval is slow in production (>1 second)
Large vector index, high-dim embeddings, no caching, synchronous pipeline, cold DB connections.
Solutions
1. ANN index (HNSW) — tune ef_search. 2. Quantize embeddings (int8 or binary). 3. Semantic query cache (Redis + GPTCache). 4. Async parallel retrieval. 5. Reduce top-k (top-5 not top-20). 6. Pre-filter with metadata before vector search.
Problem 6
Knowledge base goes stale
New documents added but embeddings not updated. Users get outdated answers. Manual re-ingestion is slow and error-prone.
Solutions
1. Incremental indexing — only embed changed/new docs. Track doc hashes. 2. Timestamp metadata + recency bias in retrieval scoring. 3. Event-driven pipeline — trigger re-embedding on doc change (Kafka, webhooks). 4. Separate "hot" index (recent) from "cold" archive, merge results at retrieval time.
Single Agents — Architecture & Patterns
Agent architectures — ReAct vs Plan-and-Execute vs others +
  • ReAct (Reason + Act) Interleave reasoning and action in a loop: Thought → Action → Observation → repeat. Most common. Good for tool-using agents with dynamic decisions. Used in LangChain AgentExecutor.
  • Plan-and-Execute Planner LLM creates full plan upfront. Executor carries it out step by step. More predictable, better for complex multi-step tasks. Can use separate cheaper model for planning.
  • Reflexion Agent self-reflects on failure, generates verbal feedback, retries. Improves on mistakes without gradient updates. Good for tasks with clear success criteria.
  • Structured output agent LLM outputs structured JSON (tool call params), deterministic execution. Less flexible but more reliable. OpenAI function calling implements this.
  • LATS (Language Agent Tree Search) Monte Carlo Tree Search for agents. Explores multiple action paths, backtracks on failure. Expensive but accurate for analytical tasks.
  • Self-RAG Agent decides when to retrieve, what to retrieve, and evaluates retrieved content quality before using it. Best quality for retrieval-heavy agents.
Agent memory types +
  • In-context (short-term) Conversation history in prompt. Fastest, limited by context window. Truncate intelligently — keep recent + summarize old.
  • External vector memory (long-term) Vector store of past interactions. Retrieve relevant history per query. Enables persistent memory across sessions.
  • Entity memory Extract and store facts about entities (people, companies) separately. Look up per entity on each turn. LangChain EntityMemory.
  • Episodic memory Store past successful task executions as examples. Retrieve similar episodes for few-shot grounding. Best for repetitive task agents.
  • Procedural memory The model weights themselves — what the agent knows how to do. Updated via fine-tuning, not prompting.
Agent failure modes and fixes +
  • Infinite loops Agent keeps calling same tool. Fix: max_iterations limit, track tool call history, detect repeated action patterns.
  • Tool hallucination Agent invents tool arguments or calls non-existent tools. Fix: strict JSON schema validation, tool call verification layer.
  • Context window overflow Long agent traces exceed context. Fix: summarize intermediate steps, external memory, scratchpad truncation.
  • Getting stuck Agent can't decide next step. Fix: explicit fallback tool, timeout + human escalation, Reflexion retry with different approach.
  • Prompt injection via tools Malicious tool output manipulates agent. Fix: sanitize tool outputs, sandboxed execution, validate actions before running.
Multi-Agent Systems
Multi-agent patterns — when to use what +
  • Orchestrator-Worker Central orchestrator delegates subtasks to specialized worker agents. Tracks progress, aggregates results. LangGraph, CrewAI. Best for defined pipelines.
  • Hierarchical Manager → team lead agents → worker agents. Scales to complex tasks. Risk: communication overhead and error propagation between layers.
  • Peer-to-peer / debate Agents interact directly, challenge each other's outputs. Better reasoning quality. Expensive. AutoGen implements this pattern.
  • Sequential pipeline Agent A output → Agent B input → Agent C input. Simple, debuggable. Good for document processing pipelines.
  • Event-driven Agents publish/subscribe to events. Loose coupling. Good for async, long-running tasks. Kafka + agent workers.
  • Mixture of Experts (routing) Router classifies task, sends to specialist agent. Each specialist optimized for its domain. No wasted compute on generalist.
Multi-agent complexities and how to tackle them +
  • State synchronization Multiple agents modifying shared state → race conditions. Fix: centralized state store (Redis, DB), optimistic locking, event sourcing.
  • Error propagation One agent fails, cascades through pipeline. Fix: checkpoint state between agents, retry with backoff, fallback paths, explicit error contracts.
  • Communication overhead Too many agent hops add latency. Fix: batch messages, compress intermediate outputs, use shared memory instead of passing full context.
  • Conflicting outputs Two agents give contradictory answers. Fix: conflict resolution agent, voting mechanism, confidence scoring per agent.
  • Cost explosion Each agent call costs tokens. Fix: small models for workers, large model only for orchestrator. Cache intermediate results aggressively.
  • Debugging Hard to trace which agent caused wrong output. Fix: full trace logging (LangSmith), unique trace IDs per agent, structured agent logs.
  • Human in the loop For high-stakes actions (send email, execute DB write), pause and get human approval. LangGraph supports interrupt() pattern natively.
Frameworks — which one to use +
  • LangGraph State machine for agents. Best for complex branching workflows needing fine control. Supports cycles, human-in-the-loop, checkpointing. Production-grade.
  • CrewAI Role-based multi-agent. Define agents with roles/goals, let them collaborate. Good for simulating team workflows. Simpler than LangGraph.
  • AutoGen (Microsoft) Conversation-based multi-agent. Agents talk to each other. Great for debate, code review patterns. Research-oriented.
  • LangChain AgentExecutor Simpler single-agent with tools. Good starting point but less control. Don't use for complex multi-hop workflows.
  • Semantic Kernel Microsoft SDK. Good for enterprise .NET + Python hybrid apps. Strong plugin/tool ecosystem.
Tool Design & Model Selection
How to design good tools for agents +
  • Single responsibility Each tool does exactly one thing. search_database() not search_and_process_and_format(). Easier to test, more reliable function calling.
  • Descriptive docstrings LLM reads tool description to decide when to use it. "Search the internal knowledge base for product documentation" beats "search docs". Description IS half the tool.
  • Typed parameters (Pydantic) Use Pydantic models for tool inputs. LLM gets schema, validation happens automatically. Never accept freeform strings where structured params work.
  • Return structured output Return JSON/dict, not raw strings. Agent can reliably parse results. Always include success/error field.
  • Limit tool count More than 10–15 tools degrades tool selection accuracy. Group related tools. Use sub-agents for specialized toolsets.
  • Observability hooks Log every tool call: inputs, outputs, latency, errors. Essential for debugging. LangSmith or custom structured logging.
Model selection — which LLM for what +
  • GPT-4o Best for complex reasoning, multi-step tool use, code generation. High function-calling accuracy. Go-to for orchestrator agents.
  • Claude 3.5 Sonnet / Opus Best for long-context tasks (200k window), instruction following, structured output. Better at NOT hallucinating when told not to. Best for faithfulness-critical RAG.
  • GPT-4o-mini / Claude Haiku Fast, cheap. Use for worker agents, simple classification, extraction tasks. Route simple queries here, complex to larger model.
  • Gemini 1.5 Pro Best for very long context (1M tokens). Use for summarizing huge doc corpora, video/audio analysis tasks.
  • Mistral / Llama 3 Open source, self-hosted. Use when data privacy/sovereignty is required. Can't send data to external API. Fine-tune on your domain.

Decision framework: Capability requirement → latency budget → cost per query → data residency constraints → pick model.

Dynamic model routing at runtime +
  • Classify query complexity (simple / medium / complex) using small classifier or rule-based check.
  • Simple (FAQ, single fact): GPT-4o-mini or Haiku. ~10ms add, 0.1¢ cost.
  • Medium (multi-step, some reasoning): GPT-4o or Sonnet.
  • Complex (code, multi-hop, long doc): GPT-4o or Claude Opus.
  • Implement with RouteLLM (open source) or custom classification layer.
  • Measure: cost per query, accuracy per tier, fallback rate. Tune thresholds on real traffic data.
Agent Evaluation & Performance
Agent evaluation metrics +
  • Task success rate Did agent complete the task correctly? Binary for simple tasks, rubric-scored for complex. Most important metric.
  • Tool call accuracy Did agent call right tool with correct params? Compare to ground truth tool calls. Measures planning quality.
  • Step efficiency How many steps to complete task? Fewer = better. Penalize infinite loops and redundant calls.
  • Trajectory similarity Compare agent action sequence to ideal trajectory. Edit distance metric. Measures reasoning quality, not just outcome.
  • Latency per task End-to-end time. Break down by LLM calls, tool calls, retrieval. Identify bottleneck per component.
  • Cost per task Total tokens × price. Track by model, task type, agent role. Required for production ROI analysis.
  • LLM-as-judge GPT-4/Claude evaluates agent output quality. Define rubric clearly. Consistent but biased toward similar model outputs. Use cautiously.
Agent latency reduction +
  • Parallelize independent tool calls If agent needs weather AND calendar, call both simultaneously with asyncio.gather(). Single biggest win for multi-tool agents.
  • Cache tool results Repeated tool calls within session return cached result. Redis with TTL. Especially valuable for DB lookups and API calls.
  • Reduce reasoning steps Better system prompt reduces unnecessary thinking. Plan-and-Execute more efficient than ReAct for predictable tasks.
  • Smaller model for planning Cheap model generates plan, expensive model only for complex execution steps. Hybrid orchestration.
  • Prompt / KV caching Cache system prompt + tools schema. Anthropic prompt caching, OpenAI context caching. Saves 85%+ on repeated prefix tokens.
  • Stream intermediate results Show users partial results as sub-tasks complete. Feels much faster even with same total wall-clock time.
Improving agent performance +
  • Better system prompt Describe agent role, constraints, output format explicitly. Include few-shot examples of correct tool use. Biggest ROI change possible.
  • Fine-tune on agent trajectories Collect successful task traces → fine-tune smaller model on them. Match GPT-4 accuracy at Llama 3 cost for domain-specific tasks.
  • Tool description quality Rewrite tool descriptions based on cases where agent chose wrong tool. A/B test descriptions. Often 10–20% accuracy lift with zero code change.
  • Constrain action space Don't give agent tools it can't use in current context. Dynamic tool injection based on task type. Fewer choices = better decisions.
  • Structured outputs Force JSON for tool selection. Remove ambiguity. OpenAI structured outputs, Outlines library for open-source models.
  • Eval-driven iteration Build eval suite first. Measure baseline. Make one change. Measure again. Never ship without evaluating. Same discipline as software testing.
Key Parameters — What Value, When & Why
LLM generation parameters
ParameterRangeWhen to use what value
temperature0.0 – 2.00.0 = deterministic, factual. Use for RAG, QA, extraction, tool-calling. 0.7 = balanced default. 0.9–1.0 = creative writing, brainstorming. Never high for agent tool-calling — they hallucinate arguments.
top_p0.0 – 1.0Nucleus sampling. Keep at 1.0 when using temperature. If using top_p as primary control, set temperature=1.0. 0.1 = very conservative. 0.95 = permissive. Don't tune both simultaneously — redundant.
max_tokensvariesSet tight based on expected output. Too high wastes money. RAG answers: 512–1024. Code gen: 2048+. Summaries: target_length × 1.3.
frequency_penalty0.0 – 2.00.3–0.5 reduces repetition of tokens. Use when output sounds repetitive. Too high → incoherent output. Keep at 0 for factual tasks.
presence_penalty0.0 – 2.00.3–0.6 encourages novel topics in brainstorming. Keep at 0 for grounded/factual tasks — you don't want the model wandering off-topic.
seedintegerSet fixed seed during evaluation runs for reproducibility. Not for production — users expect variety.
